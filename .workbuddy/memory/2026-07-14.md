## 关键发现：CI "还是不行" 的真正原因（非代码问题）
- 该仓库**历史上只跑过 1 次 Actions**（run #1，commit `6013ecd`，workflow_dispatch，failure）。用户粘贴的失败日志就是这一次的、**旧代码**产物。
- 之后所有修复提交（含 coding 端点 `16924c1`、诊断 `5e7fbd4`）都已在 origin/main，但**从未被 CI 执行过**。
- 根因：`weekly.yml` 只有 `schedule`(每日) + `workflow_dispatch`(手动) 触发，**没有 `on: push`**。所以 push 修复不会重跑 CI，用户一直看到同一条旧失败 → 误以为"没修好"。
- 解决：去 Actions 页面手动 **Run workflow** 重跑最新代码即可（配置与本地已验证的 .env 一致）。可选：加 `on: push` 让代码改动自动验证（代价：每次 push 都会消耗 ARK 额度生成整期周刊）。

## 最终根因："Verify 报 ARK_API_KEY 为空" 的真正 bug（env 作用域）
- 用户配好 GitHub Secret `ARK_API_KEY` 后 Verify 步骤仍报空。排查发现 `weekly.yml` 的 `env:` 块只挂在 **Generate Weekly** 这一 step 上（step 级），而 **Verify API config** 是独立前置 step，继承不到 `ARK_API_KEY` → 永远误判为空 → exit 1。与 secret 是否配置无关（这也是"加了还是空"的真相）。
- **修复**（commit `466fbf8`，已 commit 待手动 push）：把 `env:` 提到 **job 级别**（runs-on 之下、steps 之上），Verify 与 Generate 共享同一套变量。
- `git push origin main` 在本机仍失败（无 GitHub 凭据，/dev/tty 不可用），需用户手动推送后再 Run workflow。
- **经验**：GitHub Actions 中若多个 step 需共享 secret/变量，env 必须放 job 级或 workflow 级，不能只挂单个 step。
