## 排查：导出无图 + "采集失败：没有成功分析的资讯条目"

用户反馈两问题：① 网页"生成完整周报文档"导出的 md 看不到图片；② "新增本期资讯"按钮报
"采集失败：没有成功分析的资讯条目"。

### 根因
1. **导出无图**：`/api/export` 只回纯 md 文本，图片是相对路径 `images/...`（HTML 版用服务器
   绝对路径 `/issues/...`）。单独下载 md/html 时图片不带走、也打不开。本地 `output/weekly-2026-07-14`
   实际有图（43 张），网页内预览（/output/... 路由）正常，纯属导出格式问题。
2. **采集失败**：该错误仅在 LLM 逐条分析**全部抛异常**时抛出。本地实测 ARK Key、抓取（108 条）、
   逐条分析（3/3）均正常 → 说明用户点按钮所在的运行环境（多为部署出去的网页服务）访问不到
   `https://ark.cn-beijing.volces.com`（Key 未注入 / 无外网 / 代理问题 / 速率限制）。

### 修复
- `app/services/export_builder.py`：`_image_src_for_html` 增加 `relative` 参数（离线用相对路径）；
  `_render_overview` 封面去掉 `weekly-YYYY-MM-DD/` 前缀统一为 `images/cover.png`；
  `build_export_html` 加 `relative=False` 参数。
- `web_server.py` `/api/export`：改为返回**自包含 ZIP**（newsletter.md + newsletter.html + images/），
  mimetype=application/zip，文件名 `weekly-{date}.zip`。
- `app/pipeline.py`：`_analyze_parallel` 捕获首个异常；`run_pipeline` 在 candidates 为空时抛出的
  RuntimeError 现在包含真实异常原因 + 排查提示（Key/端点/网络/限流）。

### 验证
- ZIP 导出实测：45 条目（md+html+43 图），md 20 个图片引用全部匹配，封面引用=images/cover.png。
- 三文件 py_compile 通过，调用链无误。

  改后报错信息已能直接显示真实原因。

## 修复：Render 部署"任务已失效"

用户部署在 Render 的 `ai-weekly-report.onrender.com` 点击"新增本期资讯"后，前端轮询任务状态提示"任务已失效"。根因是 `web_server.py` 把任务状态存在内存字典 `TASKS` 里，Render 免费实例在无请求时会休眠/重启，内存清空后任务 ID 丢失。

### 修复
- `web_server.py`：引入 `output/tasks.json` 持久化任务状态；`_new_task` / `_update_task` 均写盘；
  `api_task` 内存未命中时回退读取磁盘；启动时 `_load_tasks()` 恢复历史任务；对"running 超过 30 分钟"的旧任务标记为失败（实例重启导致线程中断）。

### 验证与问题
- `web_server.py` py_compile 通过；`tests/test_newsletter.py` 13/13 通过。
- 首次实现时把 `_load_tasks()` 放在 `TASKS_FILE` 定义之前，导致 Render 部署启动失败（NameError）。已重新排序：常量 → 函数定义 → `_load_tasks()` 调用。
- 本地 `import web_server` 验证初始化成功。

## 修复：期数重复/反向、日期范围

用户反馈"新建成功"后，归档下拉显示：
- 第1期 · 2026年7月2日 — 2026年7月15日（最新一期）
- 第2期 · 2026年7月1日 — 2026年7月14日（旧一期）

最新一期反而被标成"第1期"，旧一期被标成"第2期"，期号反向。

### 根因
1. `app/pipeline.py` 生成周刊时**没有写入 `issue_number`**，字段保持默认 0。
2. `web_server.py` 的 `/api/archive` 在 `issue_number` 缺失时回退到 `(total - idx + 1)`，导致"最旧 = 最后一期、最新 = 第1期"，与直觉相反。
3. `app/services/report_builder.py` 的 `_calc_period` 与 `config.issue_period` 不一致（用 `now - 14 days` 而非 `now - 13 days`），新旧入口生成的日期区间可能差一天。

### 修复
- `app/pipeline.py`：新增 `_calc_issue_number()`，统计已有 `weekly-*/newsletter.json` 与旧 `weekly-*.json` 后 `+1`；在 `run_pipeline` 构造 `WeeklyNewsletter` 时写入 `issue_number`。
- `web_server.py`：归档回退编号改为 `idx`（最旧 = 第1期），并显式处理 `issue_number` 为 0/空的情况。
- `app/services/report_builder.py`：`_calc_issue_number` 与 pipeline 统一；`_calc_period` 改为直接调用 `config.issue_period`，保证日期区间一致。

### 验证
- `py_compile` 三文件通过；`tests/test_newsletter.py` 13/13 通过。
- 本地 Flask test_client 调用 `/api/archive`：旧一期正确显示为"第1期"；`_calc_issue_number` 返回下一期号为 2。
- 部署后，历史 `issue_number=0` 的旧文件会自动被回退编号纠正；新一期由 pipeline 写入正确期号。
